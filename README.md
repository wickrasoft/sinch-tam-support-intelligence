# TAM Support Intelligence Dashboard

A lightweight web dashboard for **Technical Account Managers** to monitor support health across dedicated enterprise accounts — built with synthetic Zendesk-style ticket data (no Zendesk admin access required for demo).

## Metrics Covered

| Metric | Description |
|--------|-------------|
| **P1 / P2 Tickets** | Critical and high-priority incident volume per account and TAM |
| **SLA Breaches** | First-response and resolution SLA violations |
| **CSAT** | Customer satisfaction scores (1–5 scale) with health indicator |
| **MTTA** | Mean Time to Acknowledge |
| **MTTR** | Mean Time to Resolve |
| **Reopenings** | Ticket reopen events filterable by day / week / month / quarter / year |

## Quick Start

```bash
npm install
npm run generate-data   # optional — regenerates dummy Zendesk dataset
npm run dev             # open http://localhost:5173
```

## Filters

- **Period**: Day, Week, Month, Quarter, Year
- **Reference Date**: anchor date for the selected period
- **TAM**: filter to one Technical Account Manager's portfolio
- **Account**: filter to a single dedicated account
- **Priority**: P1, P2, P3, or P4
- **SLA breaches only**: show only tickets that breached SLA
- **Active filter chips**: click to remove individual filters

## Dashboard Views

| Tab | Contents |
|-----|----------|
| **Overview** | KPIs with period-over-period deltas, at-risk alerts, health scores, charts |
| **Accounts** | Sortable account matrix with health score; click row to drill down |
| **Tickets** | Searchable, sortable ticket explorer with pagination |

## Enhancements

- **Account Health Score** (0–100) — composite of SLA, CSAT, reopenings, P1/P2 volume
- **At-Risk Alerts** — flags accounts needing TAM attention
- **Period Comparison** — KPI cards show ▲/▼ vs prior period (day/week/month/quarter/year)
- **Export CSV** — download filtered tickets
- **Export Report** — Markdown summary for QBR / service reviews

## Data Model

Dummy data is generated by `scripts/generateData.js` and stored in `src/data/tickets.json`. Each ticket mirrors key Zendesk fields:

- `zendesk_id`, `subject`, `priority`, `status`, `channel`
- `account_id`, `account_name`, `tam_id`, `tam_name`
- `created_at`, `first_response_at`, `solved_at`, `closed_at`
- `sla.first_response_breached`, `sla.resolution_breached`
- `mtta_minutes`, `mttr_minutes`
- `csat.score`, `reopen_count`, `reopen_events[]`

**8 enterprise accounts** across **4 TAMs**, ~1,600 tickets over 12 months.

## Zendesk Integration (Future)

When admin access is available, replace the static JSON import with Zendesk API calls:

```
GET /api/v2/search.json?query=type:ticket organization:{org_id}
GET /api/v2/ticket_metrics.json?ticket_id={id}
GET /api/v2/satisfaction_ratings.json
```

Map Zendesk priority (`urgent` → P1, `high` → P2) and metric timestamps to the existing data shape — no dashboard changes required.
